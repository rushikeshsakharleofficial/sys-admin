# Website UI Deep QA

## Requested task

$ARGUMENTS

If no explicit task is supplied, apply this skill to the current website, route, preview URL, local app, or repository context.

## Mission

Act as a strict QA engineer for a website or web app. The website may have been generated by Claude Code, Codex, Cursor, v0, Bolt, Lovable, Replit Agent, or any other AI/code generator. Assume the site may contain beautiful-looking but non-functional UI, fake buttons, placeholder links, broken responsiveness, leaked test data, poor accessibility, and unsafe browser storage.

Do not call the website passed until you have evidence.

## Non-negotiable rule

Do not randomly click around.

For every page:

1. Inspect first.
2. Create a page-specific plan.
3. Execute that plan in a controlled order.
4. Capture evidence.
5. Report exact defects.
6. State what was not tested.

## Preferred tool order

1. Use Playwright MCP for live exploratory UI testing when available.
2. Use the bundled Playwright Test files for repeatable automated testing.
3. Use source-code inspection when the repository is available.
4. Use manual instructions only for steps that cannot be safely automated.

## Safety boundaries

Never perform these without explicit user confirmation:

- payment
- booking or reservation
- public post
- sending email or messages
- destructive account changes
- deleting user data
- production deployment
- live database migration
- purchase, subscription, or billing change

Never ask the user to paste passwords, OTPs, tokens, API keys, card data, or secrets into chat. If login, 2FA, or payment is required, ask the user to take over the browser for that step.

Use fake test data unless the user explicitly provides safe test credentials or a staging environment.

## Initial assumptions to state

At the start of the report, state:

- target URL or local command used
- whether testing was unauthenticated or authenticated
- whether the environment looked like local, preview, staging, or production
- browser and viewport sizes tested
- whether destructive actions were skipped
- whether Playwright MCP, Playwright Test, or both were used

## Screenshot protocol

For every route and viewport:

1. Top viewport screenshot.
2. Full-page screenshot.
3. Scroll to 25% and screenshot.
4. Scroll to 50% and screenshot.
5. Scroll to 75% and screenshot.
6. Scroll to bottom and screenshot.
7. Return to top and screenshot if sticky/fixed UI changed.
8. Screenshot every opened modal, drawer, dropdown, popover, tooltip, toast, and floating window.
9. Screenshot every validation state.
10. Screenshot every loading, empty, error, and success state that can be reached safely.

Name screenshots predictably:

- No clipped important text.

- No content hidden under sticky headers.
- No content hidden behind sidebars.
- No content hidden behind footers.

- No layout shift after loading completes.
- No inconsistent spacing between similar components.
- No inconsistent card height where uniformity is expected.
- No broken z-index layering.
- No invisible clickable layer blocking visible controls.

## Interaction checks

For every visible interactive element:

1. Identify role, accessible name, text, selector, and location.
2. Verify visible and enabled state.
3. Hover it.
4. Capture hover screenshot.
5. Focus it with keyboard if possible.
6. Capture focus screenshot.
7. Click it only if safe.
8. Wait for UI response and network response.
9. Capture resulting screenshot.
10. Verify expected result.
11. Verify no console/page error.
12. Verify no network failure.
13. Verify no duplicate unsafe request.
14. Return to prior state if needed.

Must cover:

- all buttons
- all links
- icon-only buttons
- dropdown triggers
- menu items
- tab controls
- accordions
- toggles
- checkboxes
- radios
- selects
- date pickers
- sliders
- pagination controls
- copy/download/upload controls
- sidebar navigation
- top bar navigation
- footer links
- floating buttons
- close/minimize controls

- empty form
- invalid form

- valid form

- success state
- server error state
- network failure state if mockable

- empty string

- very long string

- accented text
- special characters
- HTML-like input: `<script>alert(1)</script>`

- invalid email
- valid email
- invalid phone
- valid phone
- negative number
- zero
- decimal
- very large number
- invalid date
- past date
- future date
- short password
- long password

### Behavior checks

- submit empty form
- validation messages appear near the correct fields
- first invalid field receives focus where expected
- errors are announced/accessibly connected where possible
- submit button disables during loading
- double click does not create duplicate submission

- Escape closes only overlays, not regular forms
- reset/cancel works
- unsaved changes warning works where expected
- file upload accepts expected types only
- password fields are masked
- password visibility toggle works
- copied values are not leaked into logs

## Authentication and session checks

When authenticated testing is available:

- login success
- login invalid credentials
- logout clears sensitive state
- protected route redirects when logged out
- back button after logout does not reveal protected data
- refresh preserves session only when expected
- expired session behavior
- unauthorized API response handling
- role-based UI visibility
- admin links hidden from non-admin users
- user switch does not leak previous user's data

Never ask the user to paste credentials. Request browser takeover for login.

## Browser storage checks

Inspect before and after key interactions:

- cookies
- localStorage
- sessionStorage
- IndexedDB
- Cache API
- service workers

Flag:

- raw passwords
- access tokens in localStorage unless explicitly intended
- refresh tokens in JS-readable storage
- PII in cookies
- missing Secure on HTTPS cookies
- missing SameSite where relevant
- auth state not cleared on logout
- protected API responses cached after logout
- stale data after user switch
- large unexpected storage blobs

## Network checks

For every page and major interaction:

- no unexpected 4xx
- no unexpected 5xx
- no CORS errors
- no duplicate API calls due to rerenders
- no infinite polling unless documented
- no excessive payloads
- no token in query string
- no password/secret in request body unless intentionally submitting to auth endpoint over HTTPS
- no secret in response
- no internal stack trace
- no database error
- no debug-only response fields in production
- no cross-user data
- no unauthorized data before login
- no protected data after logout

Scan request URLs, request bodies where safe, response headers, and response bodies.

At minimum:

- every interactive element has accessible name
- icon-only buttons have labels

- errors are associated with fields
- headings follow logical order
- landmarks exist where appropriate
- keyboard-only navigation reaches all controls

- Escape behavior works

- color contrast is not obviously poor
- no ARIA role misuse
- no duplicate IDs

Keyboard sequence:

1. Start at page top.
2. Press Tab through all controls.
3. Press Shift+Tab back through important controls.
4. Use Enter/Space on buttons and links.
5. Use arrow keys in menus/tabs/selects where expected.
6. Use Escape to close overlays.

## Severity definitions

Critical:

- app/page cannot load
- auth/session broken
- cross-user data leak
- secret/token leak
- payment/security flow broken
- data loss
- destructive action fires unexpectedly

High:

- primary navigation broken
- main CTA broken
- form cannot submit
- modal/drawer traps user
- mobile layout unusable
- protected data visible after logout
- API 5xx on core flow
- serious accessibility blocker on primary flow

Medium:

- visible overlap affects readability or use
- dropdown/popover wrong position
- validation missing
- duplicate API calls
- table unusable on one viewport
- keyboard issue on non-critical flow

Low:

- minor spacing issue
- minor text wrapping
- cosmetic inconsistency
- non-critical icon alignment
- small contrast concern

## Form automation

Run `auditForms(page)` from `helpers/forms.ts` to:

- Detect forms missing submit button
- Detect `action="#"` placeholder forms
- Flag password fields without `autocomplete`
- Flag disabled submit buttons on initial render

Run `triggerAndCaptureValidation(page, route)` to fire native HTML5 validation on empty forms and screenshot error states without submitting.

Do not submit forms to real endpoints. Use fake values only. Request browser takeover for login or payment.

## Overlay testing

Run `discoverAndAuditOverlays(page, route)` from `helpers/overlays.ts` to:

- Detect and click `aria-haspopup` triggers
- Audit open dialogs for missing `aria-modal`, `aria-labelledby`, and close button
- Test Escape dismissal
- Detect unbounded toast stacks

## SEO audit

Run `auditSeo(page)` from `helpers/seo.ts` on every public/marketing page to check:

- Title: present, length 10–80 chars, not a generic placeholder
- Meta description: present, length 50–165 chars
- Canonical link present
- Open Graph: og:title, og:description, og:image
- Twitter card
- Robots meta not accidentally set to `noindex`
- Favicon present
- Single non-empty `<h1>`
- Important images with alt text

## Security checks

Run `auditDomSecurity(page)` from `helpers/security.ts` to detect:

- Iframes without `sandbox` attribute
- `a[target="_blank"]` without `rel="noopener noreferrer"`
- Secrets in HTML comments
- Dangerous inline event handlers (`eval`, `Function(`)

Run `auditSecurityHeaders(responses)` to flag missing security headers:

- `Content-Security-Policy`
- `X-Frame-Options`
- `X-Content-Type-Options`
- `Referrer-Policy`
- `Permissions-Policy`

Run `scanUrlsForTokenLeaks(urls)` to catch tokens in query strings (JWTs, API keys, passwords).

Run `auditMixedContent(page)` on HTTPS pages to catch HTTP resource loads.

## Web Vitals

`collectPerformanceSnapshot(page)` now returns Core Web Vitals where the browser supports them:

| Metric | Good | Needs Improvement | Poor |
| --- | --- | --- | --- |
| TTFB | < 800ms | < 1800ms | ≥ 1800ms |
| FCP | < 1800ms | < 3000ms | ≥ 3000ms |
| LCP | < 2500ms | < 4000ms | ≥ 4000ms |
| CLS | < 0.1 | < 0.25 | ≥ 0.25 |

Use `poorWebVitals(snapshot)` to extract poor-rated entries. Report poor Web Vitals as Medium defects.
